# Data Schema

## Purpose

本项目的数据结构不是普通监督学习中的简单 `(x, y)` 对，而是服务于可靠优化、failure analysis 和不确定性评估的研究数据对象。换句话说，数据本身必须体现模拟电路优化场景中的约束、边界、失败和阶段来源等语义信息。只有这样，后续的代理模型训练、误判风险建模、主动采样和论文图表分析才能建立在统一且可追踪的数据基础之上。

## Sample Unit

每一条样本对应一次候选设计点的完整评估记录。样本的最小单位不是单个指标，而是一个包含设计参数、仿真结果、约束状态、边界语义与元信息的结构化对象。该对象应具备稳定字段定义，并可序列化为 CSV、JSON 或 parquet 等格式，以适应后续训练、分析和实验复现。

## Core Fields

每条样本应至少包含以下信息。第一部分是设计参数，包括原始参数值与归一化参数值，原始值用于物理解释和网表回放，归一化值用于模型训练与优化。第二部分是性能指标，即仿真器返回的 gain、GBW、phase margin、power 及其他必要输出。第三部分是约束状态，包括整体是否满足 spec，以及每一项约束是否满足。第四部分是边界信息，包括与最近约束边界的距离、是否属于边界敏感样本以及约束违背向量。第五部分是执行状态，包括是否仿真成功、失败类型、仿真耗时、缓存命中情况等。第六部分是阶段元信息，例如样本属于初始采样、主动学习补点、优化后期验证还是困难场景实验，外加随机种子、时间戳和实验配置编号。

## Constraint Violation Vector

为了支撑更精细的风险建模与 failure analysis，本项目引入约束违背向量。该向量逐项记录设计对每个约束的满足情况和偏离程度，而不是只给出一个整体可行或不可行标签。例如，一个样本可能满足 gain 和 GBW，但相位裕度低于下界；另一个样本可能所有动态指标都满足，但功耗超标。通过保留这种逐约束信息，系统可以分析模型在不同约束维度上的失效模式，并据此构造更精确的边界风险量。

## Boundary Annotation

边界标签是本数据结构的关键设计之一。对于每条样本，我们根据其相对约束面的 margin 计算边界距离，并将样本划分为 safe、boundary、infeasible 或 failed 等类别。这里的 boundary 并不是一个纯几何概念，而是服务于可靠优化研究的语义概念：它表示该样本位于“模型小误差即可改变可行性判断”的区域。此类标注为后续分析误差—边界距离关系、误判率分布以及采样价值提供基础。

## Status Semantics

样本状态必须明确区分以下几类。第一类是 success-feasible，即仿真成功且满足所有约束。第二类是 success-infeasible，即仿真成功但不满足约束。第三类是 simulation-failed，即仿真器未完成有效求解。第四类可以根据需要细分为 parse-failed、timeout 或 numerical-failed 等子类。之所以要保持这种粒度，是因为这些状态在方法和论文中具有不同含义，不能简单混用。

## Dataset Splits

训练集、验证集和测试集的划分必须在 schema 层面与实验协议保持一致。原则上，同一实验协议中不同方法共享相同的初始数据池与相同的划分方式，以保证公平性。对于主动学习或在线更新过程中新增加的数据，应保留样本来源标签，以便区分静态数据与动态补点数据，并支持后续分析不同阶段样本对模型性能和决策质量的影响。

## Storage and Versioning

数据集需要版本控制。每次重要的仿真配置变化、约束定义变化或参数空间调整，都应生成新的数据版本编号，并在元数据中记录生成条件。训练和实验脚本不得直接依赖“当前最新”数据路径，而应通过配置文件显式指定数据版本。这样做的目的，是保证任何论文图表和结果都能追溯到明确的数据来源。

## Design Principle

本项目的数据 schema 不是附属工程，而是论文方法不可分割的一部分。因为 failure mode 是否存在、不确定性是否有意义、边界风险是否真实、主动采样是否有效，最终都需要通过样本级结构化字段加以验证。数据结构如果设计粗糙，后续所有高层方法都会失去支撑。因此，本项目将数据 schema 视为连接仿真系统、代理模型和评测体系的中枢。
