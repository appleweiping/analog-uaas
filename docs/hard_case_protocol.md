# Hard Case Protocol

## Purpose

困难场景实验的目的并不是让方法在更多设置上“看起来更全面”，而是有意识地构造最容易触发代理模型 failure mode 的任务，使 baseline 的弱点被放大，从而检验本文方法是否真正解决了核心问题。顶会论文中，平均指标往往不如关键场景下的系统性优势更有说服力。对于本项目而言，hard cases 的价值在于：它们能够迫使优化器频繁接近约束边界，使误判风险成为决定成败的关键因素。

## Design Principle

所有困难场景都必须满足两个原则。第一，它们必须仍然是合理的模拟电路设计任务，而不是脱离物理背景的极端数值设置。第二，它们必须与本文定义的 failure mode 有直接联系，也就是说，它们应增加边界区样本比例、强化 trade-off 冲突或收缩可行域，从而更容易暴露代理模型的后期失效问题。

## Candidate Hard Cases

第一类困难场景是 tightened phase margin setting，即显著提高相位裕度下界，使优化过程更频繁地在稳定性边界附近活动。这类设置直接针对本文最容易分析的 boundary failure。第二类困难场景是 low-power high-GBW conflict setting，即同时收紧功耗上限并提高 GBW 目标，从而制造明显的多目标冲突，使简单 surrogate-only 方法更容易被局部误判牵引。第三类困难场景是 multi-constraint tight setting，即同时逼近 gain、GBW 和 power 等多个约束面，使可行域大幅缩小，增强 false feasible 的危害。必要时，还可以加入对负载条件或供电条件的小幅变化，以考察方法在困难约束下的稳健性。

## Experimental Setup

每个困难场景都应拥有独立配置文件，并与主实验使用相同的初始数据版本、相同的随机种子集合和相同的仿真预算上限。这样做的目的是保证比较公平性，并确保观察到的性能差异主要来自方法在高风险区域的决策能力，而不是由数据、预算或初始化差异造成。

## Evaluation Focus

困难场景实验不以平均误差为主要评价标准，而以以下问题为核心：哪些方法能够在有限仿真预算下找到可行解；哪些方法在后期会频繁误判为可行并反复调用无效仿真；哪些方法能在高风险边界区保持更稳定的轨迹。应重点报告成功率、达到 spec 所需的仿真次数、误判事件频率以及后期轨迹震荡度。

## Role in the Paper

hard-case protocol 在论文中的作用，是把“本文方法平均上更好”升级为“本文方法在最关键、最脆弱的场景下依然可靠”。对于围绕 reliability 展开的工作来说，这部分不是锦上添花，而是必要证据。因为一个在容易场景中表现良好的方法，并不能说明它具备可靠优化能力；只有在高风险设置下仍能稳定工作，方法的研究价值才真正成立。
